# 1. Verbindliches fachliches Ziel

TM01B bildet eine dauerhaft identifizierte strategische Konvoientität ab, die nur in relevanten Gebieten als physische DCS-Gruppe existiert.

```text
außerhalb eines Reveal-Fensters = virtuelle Entität
innerhalb eines Reveal-Fensters = physische DCS-Gruppe
```

Die reguläre Bedienung besteht aus genau einem Startbefehl:

```text
F10 → OMW Tests → TM01B → Start convoy
```

Danach müssen virtuelle Bewegung, Materialisierung, physische Fahrt, Dematerialisierung und Zielankunft automatisch ablaufen.

# 2. Heute bestätigte Designentscheidungen

## 2.1 Ein Kreis pro Reveal-Fenster

Pro sichtbarem Abschnitt wird genau eine kreisförmige Mission-Editor-Triggerzone verwendet:

```text
ZONE_TM01_REVEAL_01
ZONE_TM01_REVEAL_02
```

Der Kreis selbst definiert den vollständigen Sichtbereich:

```text
innerhalb des Kreises  = physisch sichtbar
außerhalb des Kreises  = virtuell
```

Separate Entry- und Exit-Zonen sind für dieses Zielmodell nicht erforderlich.

Die Fahrtrichtung ist für die Sichtbarkeitsentscheidung irrelevant. Sie wird nur für virtuelle Bewegung, Spawnaufstellung, Fahrzeug-Heading und physische Route benötigt.

## 2.2 Virtueller Kartenmarker

Während einer virtuellen Phase soll BLUE einen beweglichen Marker sehen. Dieser soll mindestens anzeigen:

- berechnete virtuelle Position;
- nächstes Reveal-Fenster beziehungsweise Ziel;
- ETA;
- Fortschritt der aktuellen virtuellen Etappe.

Während der physischen Phase wird der virtuelle Marker entfernt.

## 2.3 Atomare Gruppenmaterialisierung

Eine DCS-Gruppe wird atomar erzeugt. Deshalb müssen beim Materialisieren alle aktuell benötigten Fahrzeugslots gleichzeitig an gültigen Positionen erzeugt werden.

Für einen Straßenkonvoi wird die Straße selbst als Spawngeometrie verwendet. Eine beliebige freie Fläche neben der Straße ist kein bevorzugtes Modell, weil die Gruppe anschließend möglicherweise nicht auf die Straße einfädeln kann.

## 2.4 Individuelle Fahrzeugpositionen

Jeder Fahrzeugslot erhält:

- eine eigene absolute Position;
- ein eigenes lokales Heading;
- einen definierten Abstand zum vorausfahrenden Fahrzeug.

Ein starres Gruppen-Template mit nur einer Verschiebung und einer gemeinsamen Ausrichtung ist für gekrümmte Straßen nicht ausreichend.

## 2.5 Fail-closed

Kann keine vollständige und sichere Aufstellung ermittelt werden, darf kein unsicherer Ersatzspawn erfolgen.

```text
kein gültiges Layout
→ kein Spawn
→ präziser Fehlerzustand
```

## 2.6 Laufzeitbudget

Die Spawnplanung darf nicht fortlaufend in jedem Simulationstick ausgeführt werden.

Verbindliche Strategie:

- bekannte Reveal-Fenster: beim Missionsstart einmal planen und cachen;
- dynamische unbekannte Spawns: begrenzte Kandidatensuche beim Aktivierungsereignis;
- maximal eine lokale Straßenpfadberechnung pro Kandidat;
- keine Straßenpfadberechnung pro Fahrzeug;
- Abbruch beim ersten gültigen Kandidaten;
- harte Obergrenze für Kandidaten und Suchdauer;
- bei vielen gleichzeitigen Anforderungen Planung über mehrere Scheduler-Zyklen verteilen.

# 3. Bestätigte MOOSE-Fähigkeiten und Grenzen

Die vendorte MOOSE-Version stellt die notwendigen Bausteine bereit:

```text
COORDINATE:GetClosestPointToRoad()
COORDINATE:GetPathOnRoad()
SPAWN:InitSetUnitAbsolutePositions()
COORDINATE:WaypointGround(..., "On Road")
```

Damit kann der Controller:

1. einen Straßenpunkt bestimmen;
2. einen Straßenpfad berechnen;
3. pro Einheit eine absolute Position und Ausrichtung setzen;
4. der Gruppe anschließend eine On-Road-Route zuweisen.

MOOSE besitzt jedoch keinen einzelnen Aufruf mit der Semantik:

```text
Spawne diese komplette Gruppe garantiert kollisionsfrei auf einer Straße.
```

Die sichere Kandidatenauswahl, Oberflächenprüfung, Scenery-Prüfung, Layoutprüfung und Recovery-Logik bleiben Verantwortung unseres Controllers.

# 4. Chronologie der Versuche

## 4.1 Version 4 – getrennte Entry-/Exit-Zonen

### Architektur

Version 4 verwendete pro Reveal-Abschnitt zwei Zonen:

```text
REVEAL_ENTRY
REVEAL_EXIT
```

Die virtuelle Bewegung endete am Entry-Zentrum. Dort wurde die vollständige Gruppe erzeugt. Die physische Route führte über konfigurierte Routenanker bis zur Exit-Zone. Dematerialisiert wurde, nachdem jeder überlebende Slot die Exit-Zone irgendwann passiert hatte.

### Mock-Ergebnis

Der gemockte Lauf bestand:

- Bootstrap `READY`;
- zwei automatische Materialisierungen;
- zwei automatische Dematerialisierungen;
- sequentielle Exit-Passage ohne gleichzeitige Anwesenheit aller Fahrzeuge;
- native Destroy-Bestätigung;
- virtuelle Zielankunft genau einmal.

### DCS-Ergebnis

Das erste Reveal-Fenster funktionierte im DCS-Lauf gut.

Das zweite Reveal-Fenster wurde erreicht und Generation 2 wurde mit sechs Fahrzeugen erzeugt. Die Gruppe blieb jedoch in `PHYSICAL_MOVING` stehen und erreichte die Exit-Zone nicht.

Sichtbeobachtung:

- die letzten beiden Fahrzeuge standen im Wasser beziehungsweise Fluss;
- die gesamte Gruppe bewegte sich danach nicht.

Logbeobachtung:

```text
physicalGeneration=2
movementState=PHYSICAL_MOVING
lastError=none
exitSeenSlotCount=0
```

### Ergebnis

Version 4 bewies die Zustandsmaschine und automatische Umschaltung grundsätzlich. Sie bewies nicht die robuste physische Platzierung. Das starre Template am Entry-Zentrum war ungeeignet.

## 4.2 Diskussion nach Version 4

Aus dem DCS-Lauf wurden drei Erweiterungen abgeleitet:

1. beweglicher Marker für die virtuelle Konvoiposition;
2. ein vollständiger kreisförmiger Sichtbereich pro Fenster;
3. individueller Straßenspawn aller Fahrzeuge.

Zusätzlich wurde klargestellt:

- Der Kreis benötigt keine Richtungsinformation.
- Alles innerhalb des Kreises ist sichtbar, alles außerhalb virtuell.
- Die Routenrichtung bleibt nur für Bewegung und Aufstellung relevant.

## 4.3 Version 5 – kreisförmige Fenster und Straßenpfad

### Architektur

Version 5 führte ein:

- eine kreisförmige Triggerzone pro Fenster;
- globalen MOOSE-Straßenpfad durch Start, sieben Anker und Ziel;
- virtuellen Moving Marker;
- automatische Bestimmung der Kreisschnittintervalle;
- individuelle absolute Fahrzeugpositionen;
- individuelle lokale Headings;
- Fail-closed bei angeblich ungültigem Straßenlayout.

### Mock-Ergebnis

Der lokale Mock-Lauf bestand:

- zwei kreisförmige Fenster erkannt;
- Moving Marker entlang eines simulierten Straßenpfades;
- zwei Materialisierungen und Dematerialisierungen;
- sechs Positionen pro Spawn;
- Zielankunft.

### DCS-Ergebnis

Version 5 lud korrekt:

```text
configurationVersion=TM01B-controlled-caching-5
bootstrap_outcome=READY
roadAlignedAbsoluteVehicleSpawn=true
virtualMarkerEnabled=true
```

Der Lauf stoppte jedoch vor dem Start der Bewegung:

```text
convoy_route_plan_failed
reason=planned vehicle position is not on road: ZONE_TM01_REVEAL_01
movementState=AUTOMATION_FAILED
physicalGeneration=0
```

### Ursache

Der globale Pfad wurde in 20-Meter-Schritte interpoliert. Diese interpolierten Punkte wurden anschließend erneut gegen die nächstgelegene Straße geprüft. Eine Abweichung über drei Meter führte zum Gesamtabbruch.

Auf engen Kurven oder ungenauen Afghanistan-Straßengraphen konnte ein aus dem Straßenpfad abgeleiteter Punkt die sekundäre Drei-Meter-Prüfung nicht bestehen.

### Ergebnis

Der Fail-closed-Mechanismus verhinderte einen unsicheren Spawn, aber die Validierung war falsch kalibriert. Moving Marker und physischer Version-5-Spawn wurden in diesem Lauf nicht ausgeführt.

## 4.4 Version 5.1 – Toleranz- und Sampling-Hotfix

### Änderung

```text
routeSampleMeters             20 m → 5 m
physicalWaypointSpacing      250 m → 100 m
roadPositionTolerance          3 m → 25 m
vehicleSpacing                18 m → 15 m
spawnInteriorMargin           12 m → 25 m
physicalClearance             40 m → 75 m
```

### DCS-Ergebnis

Version 5.1 lud korrekt:

```text
configurationVersion=TM01B-controlled-caching-5.1
bootstrap_outcome=READY
```

Die Planung stoppte erneut vor der Bewegung:

```text
convoy_route_plan_failed
reason=reveal window exit is too close to route target: ZONE_TM01_REVEAL_02
movementState=AUTOMATION_FAILED
physicalGeneration=0
```

### Ergebnis

Der Hotfix beseitigte nicht den Gesamtfehler. Er verschob nur den zuerst sichtbaren Blocker.

# 5. Nach Version 5.1 identifizierte Systemfehler

## 5.1 Fehlende Terminal-Window-Logik

Das letzte Reveal-Fenster kann nahe am globalen Ziel liegen. Der aktuelle Controller legt `virtualResumeDistance` und `physicalRouteEndDistance` hinter den Kreisaustritt. Liegen diese Werte hinter dem Routenziel, wird die Planung abgebrochen.

Nur die Abbruchprüfung zu entfernen wäre ebenfalls falsch: Danach könnte eine virtuelle Etappe von einer Position hinter dem Ziel zurück zum Ziel entstehen und wegen nicht positiver Distanz scheitern.

Erforderlich:

```text
letztes Fenster verlassen
→ verbleibende positive Reststrecke virtuell fahren
oder
→ unmittelbar ARRIVED
```

## 5.2 Nächstgelegene Straße wird nur geprüft, nicht verwendet

Der aktuelle Code berechnet `nearestRoad`, spawnt aber weiterhin auf dem ursprünglichen interpolierten `vec2`.

Damit bedeutet eine Toleranz von 25 Metern lediglich:

```text
Eine Straße ist höchstens 25 m entfernt.
```

Es bedeutet nicht:

```text
Das Fahrzeug steht auf dieser Straße.
```

Die finale Spawnposition muss aus `nearestRoad:GetVec2()` stammen.

## 5.3 Keine Wasserprüfung

Der Controller prüft derzeit nicht:

- `land.getSurfaceType()` an der finalen Position;
- Wasser beziehungsweise flaches Wasser;
- Flussbett oder ungeeignete Uferpositionen;
- alternative Layouts innerhalb desselben Fensters.

Die Version-4-Beobachtung mit zwei Fahrzeugen im Fluss ist daher weiterhin nicht technisch ausgeschlossen.

## 5.4 Kein echter Kreismarker auf der F10-Karte

Der aktuelle Code erzeugt nur einen Textmarker am Zonenmittelpunkt und schreibt den Durchmesser in den Text.

Das ist keine sichtbare Kreisgrenze. Eine echte Visualisierung des Sichtfensters muss separat über eine geeignete Map-Drawing-/Markierungsfunktion erfolgen oder bewusst als nicht dargestellt dokumentiert werden.

## 5.5 Materialisierung erfolgt nicht exakt am Kreiseintritt

Der aktuelle Triggerpunkt für die Materialisierung liegt um Formationslänge und Innenabstand hinter dem Eintritt. Der virtuelle Konvoi ist damit bereits innerhalb des Sichtfensters, bevor die physische Gruppe erscheint.

Die gewünschte Semantik lautet dagegen:

```text
virtueller Referenzpunkt erreicht Kreis
→ physische Gruppe erscheint
```

Die Aufstellung darf dabei rückwärts entlang der Straße innerhalb des Kreises erfolgen.

## 5.6 Kein Bewegungs-Watchdog

Wenn DCS die Route akzeptiert, die Gruppe sich aber nicht bewegt, bleibt der Controller unbegrenzt in `PHYSICAL_MOVING`.

Erforderlich:

- Referenzposition und Geschwindigkeit nach Spawn speichern;
- nach definierter Anlaufzeit Bewegung prüfen;
- minimale Wegstrecke innerhalb eines Zeitfensters verlangen;
- bei Stillstand Gruppe kontrolliert entfernen;
- nächsten vorbereiteten Spawnkandidaten versuchen oder präzise abbrechen.

## 5.7 Unwahre Erfolgsfelder

Felder wie

```text
allTemplateSlotsPlacedOnRoad=true
```

sind momentan nicht gerechtfertigt, weil nur die Nähe zu einer Straße geprüft wird. Erfolgsfelder dürfen erst nach Prüfung der tatsächlich verwendeten finalen Positionen gesetzt werden.

# 6. Bewertung der diskutierten Alternativen

## 6.1 Exakte Position und Ausrichtung jeder Einheit

**Bewertung: bevorzugter Ansatz für Straßenkonvois.**

Dies funktioniert auch für dynamische, vorher unbekannte Spawns. Der Aktivierungsgrund – Spieler, Feind, Sichtbereich oder anderes Ereignis – liefert lediglich eine Suchregion. Der Planner bestimmt daraus zur Laufzeit einen geeigneten Straßenabschnitt.

## 6.2 Große freie Fläche neben einer Straße suchen

**Bewertung: für Konvois nicht bevorzugt.**

Mögliche Nachteile:

- Gruppe muss erst auf die Straße einfädeln;
- Mauern, Gebäude, Gräben oder Ufer können dazwischen liegen;
- DCS-Formation kann sich gegenseitig blockieren;
- erfolgreiche Platzierung garantiert keine erfolgreiche Routenaufnahme.

Dieser Ansatz bleibt für stationäre Lager, Artillerie, SAM-Stellungen oder Infanterie relevant, nicht als Standard für Straßenkonvois.

## 6.3 Sehr kompakte Gruppenvorlage

**Bewertung: unterstützende Maßnahme, keine vollständige Lösung.**

Vorteile:

- kürzerer benötigter Straßenabschnitt;
- kleinerer Gruppenfußabdruck;
- geringere Kollisionswahrscheinlichkeit.

Nicht gelöst werden:

- Kurven;
- unterschiedliche Fahrzeugheadings;
- Wasser- oder Scenerypunkte;
- starres Layout neben der Straße;
- Bewegungsblockaden.

# 7. Vereinbarter Zielentwurf für die Fortsetzung

## 7.1 RoadSpawnPlanner

Ein eigenständiger Planner erhält mindestens:

```text
Straßenpfad oder Suchregion
Fahrzeuganzahl
Fahrzeugabstand
bevorzugte Fahrtrichtung
Reveal-Zone
maximale Kandidatenanzahl
```

Er liefert:

```text
Position und Heading je Template-Slot
lokale physische Route
Kandidaten-ID
Validierungsdaten
```

oder einen präzisen Fehler.

## 7.2 Kandidatensuche für bekannte Fenster

Für jedes bekannte Reveal-Fenster:

1. Straßenintervall innerhalb des Kreises bestimmen;
2. mehrere mögliche Lead-Distanzen innerhalb dieses Intervalls erzeugen;
3. Fahrzeuge rückwärts entlang des Pfades verteilen;
4. jeden Rohpunkt auf die tatsächlich nächstgelegene Straße projizieren;
5. finale Straßenpunkte und Headings neu berechnen;
6. Oberfläche, Abstand, Reihenfolge und Kreiszugehörigkeit prüfen;
7. gültige Layouts cachen;
8. ersten Kandidaten verwenden, weitere als Recovery-Kandidaten behalten.

Diese Planung erfolgt einmal beim Missionsstart, nicht beim Eintritt des virtuellen Konvois.

## 7.3 Dynamische unbekannte Spawns

Für spätere Systeme mit Spieler- oder Feindnähe:

1. begrenzten Entfernungsring beziehungsweise Sektor erzeugen;
2. maximal 8–12 Kandidaten auf nahe Straßen projizieren;
3. pro Kandidat höchstens einen kurzen lokalen Straßenpfad berechnen;
4. vollständiges Gruppenlayout prüfen;
5. beim ersten gültigen Kandidaten abbrechen;
6. Ergebnis cachen;
7. bei hoher Last Planungen zeitlich staffeln.

## 7.4 Oberflächen- und Kollisionsprüfung

Mindestens erforderlich:

- finaler Punkt stammt aus der Straßenprojektion;
- Wasser und ungeeignete Oberflächentypen ablehnen;
- Mindestabstand zwischen Fahrzeugen;
- vollständige Formation innerhalb des Reveal-Kreises;
- geordnete Positionen entlang des Pfades;
- keine Überlappung mit bekannten Einheiten und Statics;
- Sceneryprüfung soweit die DCS-API belastbare Daten liefert.

Eine absolute Garantie gegen fehlerhafte Kartenscenery ist nicht realistisch. Das System muss deshalb zusätzlich einen Bewegungs-Watchdog besitzen.

# 8. Priorisierte nächste Schritte

## P0 – vor jedem weiteren DCS-Test

1. Version 5.1 ausdrücklich als nicht abnahmefähig markieren.
2. Terminal-Window- und Zielankunftslogik korrigieren.
3. Tatsächliche `nearestRoad`-Koordinate als Spawnposition verwenden.
4. Wasser-/Oberflächenprüfung ergänzen.
5. Kandidatensuche innerhalb des Reveal-Fensters implementieren.
6. Bewegungs-Watchdog und Recovery-Kandidat implementieren.
7. irreführende Erfolgsfelder und Dokumentation korrigieren.

## P1 – Sichtbarkeit und Diagnose

1. echten F10-Kreis oder alternative visuelle Fensterdarstellung implementieren;
2. finale Position jedes Slots mit Koordinate, Oberflächentyp und Kandidaten-ID loggen;
3. Planungsdauer und Kandidatenanzahl messen;
4. Marker-Updatekosten messen;
5. DCS-Laufzeitprofil für mehrere gleichzeitige Planner-Anfragen vorbereiten.

## P2 – Abnahme

Ein neuer DCS-Lauf ist erst sinnvoll, wenn P0 vollständig umgesetzt ist.

Der Lauf muss nachweisen:

- Bootstrap `READY`;
- globaler Straßenpfad kompiliert;
- Moving Marker bewegt sich und ETA sinkt;
- beide kreisförmigen Fenster werden erreicht;
- alle sechs Fahrzeuge stehen nachweislich auf finalen Straßenpunkten;
- keine Fahrzeuge stehen im Wasser oder in Scenery;
- Gruppe nimmt innerhalb der Watchdog-Frist Bewegung auf;
- Fenster 1 wird vollständig verlassen und dematerialisiert;
- virtuelle Zwischenetappe läuft;
- Fenster 2 wird vollständig verlassen und dematerialisiert;
- Terminal-Window führt korrekt zu `ARRIVED`;
- Verluste werden zwischen Generationen erhalten;
- nie zwei physische Generationen gleichzeitig;
- `convoy_route_arrived` genau einmal.
